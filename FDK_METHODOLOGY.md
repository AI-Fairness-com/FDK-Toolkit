This document is the complete, public evidence trail for every fairness metric implemented across

grounded in the peer-reviewed paper that defines its criterion, or is a transparent, documented
extension of validated components into a capability gap the literature does not directly name.

## 1. How This Audit Was Conducted

### Phase A — Ground-truth extraction
Every metric was extracted directly from the 7 pipelines' actual source code (not from documentation,
not from prior published figures) by parsing every `metrics[...] = ...` assignment and resolving all
dynamic/loop-generated keys by hand. This produced a verified real count per domain, cross-checked
against the internal config dictionaries and, where discrepancies were found, against the literal
returned-metrics dictionary the pipeline produces at runtime.

**Result: 158 total metric-slots across the 7 domains (some metrics recur across multiple domains).**
The previously published figure of 89 was found to significantly undercount the toolkit's actual
metric coverage.

### Phase B — External citation search
For each of the 158 metrics, a citation search was conducted against:
- **IBM AI Fairness 360 (AIF360)**, **Microsoft Fairlearn**, and **scikit-learn** — official API documentation
- **Peer-reviewed academic literature** — searched and verified per metric, not assumed from memory

A match required the same underlying **formula**, not merely a similar name.

### Phase C — Formula-level forensic review
For every metric that did not resolve cleanly in Phase B, the actual source code was read line-by-line
and its exact mathematical formula was extracted and tested against three questions:
1. Is it a literal, transparent combination of metrics already confirmed in Phase B?
2. Is it the same statistical technique as an already-confirmed metric, applied to a different (but
   still valid) base measure?
3. Does the formula actually measure what its name claims?

This phase surfaced **6 genuine implementation defects** — cases where the answer to question 3 was
no. These were not citation gaps; they were code that needed to be fixed. All 6 have since been
corrected and re-verified (see Section 4).

### Phase D — Peer-review completeness pass
Following Phase C, every metric's citation was re-examined against a stricter standard: does it carry
at least one reference to a genuinely peer-reviewed manuscript, conference paper, or academic book —
not merely a link to official library documentation? This surfaced two further categories of problem,
both corrected:

- **Weak-source citations.** 15 metrics had been cited to Wikipedia; 13 (5 distinct papers) had been
  cited to arXiv preprints whose peer-reviewed venue was not linked, even where that venue was already
  named in the surrounding text. Several others relied on author personal-page-hosted PDFs rather than
  the publisher's own copy. All were replaced with the verified official publisher, conference
  proceedings, or journal link — see Section 6.
- **Docs-only citations.** ~76 metrics carried only an official AIF360/Fairlearn/scikit-learn
  documentation link, with no separate academic paper attached. Each was individually re-examined and
  assigned to one of two outcomes: (a) grounded in the specific peer-reviewed paper that defines its
  fairness criterion (the large majority), or (b) confirmed as a genuinely elementary statistical
  primitive (raw counts, rates, or standard ML performance measures) for which official documentation
  is the appropriate and sufficient citation, since forcing an unrelated academic citation onto basic
  arithmetic would itself be a credibility weakness, not a strength.

### Phase E — Cross-library verification for the remaining original compositions
A small number of metrics (6) remained grounded only in the *technique* they apply (e.g., the max-min
group-disparity method), not in a paper defining the specific domain application. Rather than accept
this, every other fairness library named in the reference manuscript — Themis-ML, Google's What-If
Tool, Aequitas, DoWhy, Google's Fairness Indicators, and Microsoft's Responsible AI Toolbox — was
specifically checked against these 6 metrics, to confirm no equivalent named metric existed elsewhere
that had simply been missed. None did. One metric (`causal_pathway_disparity`) surfaced a genuine,
relevant academic literature (path-specific causal fairness, associated with DoWhy) — but that
literature was deliberately **not** cited, because FDK's actual implementation does not perform the
causal/mediation analysis that literature describes, and doing so would overclaim the metric's rigor
rather than accurately document it. See Section 5 for the complete, honest accounting of these 6.

## 2. Why FDK Has More Metrics Than Any Single Library Provides

IBM AIF360 and Microsoft Fairlearn are general-purpose classification-fairness toolkits. They provide
the mathematical primitives (rate differences, ratios, calibration measures, subgroup-discovery
algorithms) but were not built with a business, education, finance, governance, health, hiring, or
justice *lens* — they do not encode domain judgment about which comparisons matter in a specific
regulated context, nor do they address fairness monitoring over time, cross-validation robustness, or
pre-audit data-quality checks.

FDK's approach: **the large majority of the 158 metrics are these library primitives directly, or the
identical formula relabeled for domain meaning** (e.g., Health's "Overtreatment Disparity" is literally
AIF360's `false_positive_rate_difference`), each independently grounded in the peer-reviewed paper that
defines its criterion — Dwork et al., Feldman et al. (statistical parity/disparate impact), Hardt,
Price & Srebro (equalized odds/equal opportunity), Berk et al. (overall accuracy equality, treatment
equality), Chouldechova (predictive parity), Pleiss et al. and Guo et al. (calibration), Kusner et al.
(counterfactual fairness), Zemel et al. (individual fairness), Hashimoto et al. (worst-case group
performance), Agarwal, Dudik & Wu (fair regression / bounded group loss), Kohavi (cross-validation
methodology), Efron (bootstrap resampling), Brodersen et al. (balanced accuracy), Breiman (permutation
feature importance), Gini via Ceriani & Verme (inequality concentration), Corbett-Davies et al.
(predictive equality, conditional statistical parity), Zafar et al. (disparate mistreatment), Foulds et
al. (differential fairness), and Wasserman (foundational statistical dispersion measures).

The remainder extend a validated technique into a capability gap neither AIF360 nor Fairlearn address
at all — composite scores built transparently from validated components (grounded in the OECD/JRC
handbook on composite indicator methodology), temporal/cross-validation robustness checks, and a small,
explicitly-flagged set of FDK-original domain compositions (Section 5).

## 4. The 6 Defects Found and Fixed

These required a code fix, not a citation — each is documented here with the exact problem, the fix
applied, and the verification performed before this document was published.

### Business — 3 fabricated-temporal metrics
`temporal_fairness_consistency`, `long_term_outcome_parity`, `dynamic_policy_fairness` previously used
a random 50/50 split of a single-snapshot dataset as a stand-in for real time periods (an explicit
comment in the original code admitted this), and silently returned a hardcoded optimistic score
(0.9–0.95) whenever data was insufficient.

**Fix**: all three now check for a genuine `timestamp` column, sort chronologically, bin into real
daily windows, and compute already-validated metrics (composite bias score, base rate, per-group
selection rate) across those genuine time periods. When no timestamp data is available, they now
return `None` rather than a fabricated passing score.

**Verified**: functional tests confirm correct `None` output with no timestamp column, and correct
real-valued output when genuine timestamped data spanning multiple days is supplied. Live-verified in
the public repository.

### Education — 2 defects
- **`educational_mobility_index`** previously correlated group performance against `range(len(groups))`
  — an arbitrary sequential index with no relationship to actual group advantage or privilege.
  **Fix**: now requires an explicit, optional `advantage_rank` input column representing genuine
  relative advantage; returns `None` when that column is not supplied, rather than a meaningless
  correlation coefficient.
- **`model_explanation_parity`** was a hardcoded stub (`= 0.0`, commented "Placeholder for speed") —
  never actually computed. **Fix**: now computes genuine per-group mean-absolute feature–prediction
  correlation (the same SHAP-adjacent proxy technique already validated for `feature_attribution_parity`),
  and reports the max–min disparity of that value across groups.

**Verified**: functional tests confirm both metrics return `None` when their required input is absent,
and real, non-trivial float values when it is present. Live-verified in the public repository.

### Governance — 1 defect
**`missingness_bias_index`** previously computed a single global average of missing-data rate across
all columns — never conditioned on group, and therefore structurally incapable of detecting
*differential* missingness, which is the only thing that would justify calling it a bias metric.

**Fix**: now computes the missingness rate independently per group for each feature, then reports the
max–min disparity across groups — directly measuring differential missingness.

**Verified**: a functional test with deliberately equal missingness across groups returns a near-zero
score (0.02); an identical *overall* missingness rate but concentrated entirely in one group correctly
returns a high score (0.8) — demonstrating the fix now distinguishes exactly the case the original
formula could never detect. Downstream consumption of this metric (`composite_governance_fairness_index`)
was re-verified to still function correctly. Live-verified in the public repository.

## 5. The 6 Metrics With Technique-Level (Not Concept-Level) Backing — Full Honest Accounting

Every one of the 158 metrics carries at least one peer-reviewed reference. For 152 of them, that
reference defines the specific fairness concept the metric measures. For the following 6, the citation
present covers the underlying *statistical technique* (the max-min or ratio group-disparity method,
grounded in Feldman et al. 2015), not a paper that independently names the specific domain composition.
This section exists so that distinction is never buried in a spreadsheet cell — it is stated here,
explicitly, as the single most important caveat in this entire registry:

| Metric | Domain | What it computes | Why no concept-specific paper is cited |
|---|---|---|---|
| `causal_pathway_disparity` | Education | (predicted rate − actual rate) gap per group, max–min | Real "path-specific effect" literature exists (Nabi & Shpitser; associated with DoWhy) but requires causal-graph/mediator analysis this metric does not perform. Citing that literature would overclaim the metric's rigor. |
| `educational_mobility_index` | Education | Correlation of a real advantage-rank input against group performance | No equivalent named metric found in AIF360, Fairlearn, Themis-ML, What-If Tool, Aequitas, DoWhy, Fairness Indicators, or Responsible AI Toolbox |
| `learning_trajectory_fairness` | Education | Max–min disparity of per-group variance in a longitudinal progress-score column | Same — checked against all libraries above, no match |
| `opportunity_access_parity` | Education | Min/max ratio of (selection rate ÷ base rate) per group | Structurally the same technique as `disparate_impact` (Feldman et al. 2015), but no library or paper names this specific composed ratio |
| `opportunity_gap_metric` | Education | Max–min disparity across resource/access-labeled feature columns | Same — no independent literature |
| `risk_stratification_fairness` | Health | Max–min disparity of per-group predicted-probability standard deviation | Same — no independent literature |

**These 6 are original FDK compositions.** That is not, on its own, a weakness — it is exactly the kind
of domain-specific extension Section 2 describes as FDK's actual contribution beyond wrapping existing
libraries. What matters is that this document says so plainly, rather than implying an external
validation that does not exist.
